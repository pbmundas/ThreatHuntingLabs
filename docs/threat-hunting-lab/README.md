# Blacksite Browser Threat Lab

A static GitHub Pages platform where administrators publish predefined investigations and learners only investigate assigned evidence.

## Open locally

Do not double-click `index.html`; browsers block WebAssembly workers and local evidence requests on `file://` pages. Double-click `start-lab.cmd` instead, or serve this directory over HTTP and open the printed local URL.

## Roles

- **Administrator:** keeps raw logs and plain-text answers private, prepares a private lab spec, runs the publishing tool, reviews the generated Parquet data and manifest, then deploys the static site.
- **Learner:** selects a published lab, reads its hypothesis and challenges, queries its assigned evidence with SQL or KQL, and submits answers. Learners cannot upload or replace evidence through the UI.

## Local learner profiles

Learners register and sign in with a device-local profile. A salted PBKDF2 password verifier and per-user challenge progress are stored in that browser's local storage. This works on GitHub Pages without a backend, but profiles do not sync across browsers or devices and are not a replacement for centralized authentication. Learners should use a non-sensitive password.

The included range contains nine OTRF investigations with five challenges each. Cases cover HTTP listener exposure, BITSAdmin download, WMIC XSL script processing, Mimikatz/LSASS access, SMBExec service execution, Regsvr32 proxy execution, elevated scheduled-task persistence, remote WMIC account creation, and PowerShell Remoting. The evidence mixes Windows Security, Sysmon, PowerShell, Task Scheduler, Service Control Manager, WMI, network-filtering telemetry, and parsed packet-capture flows where available.

The login screen is shown first, with separate **Login** and **Register** actions. Registration includes a **Back to login** action. Inside an investigation, **All labs** in the top bar returns to the difficulty-sorted catalog even when the challenge sidebar is hidden.

## SIEM-style investigation workspace

- Every returned event is displayed as parsed field/value columns with a synthesized `raw_event` field.
- **VIEW** opens an event inspector containing all populated and null parsed fields plus the original raw message or JSON event.
- Hunt statistics profile returned events by time, event category, log source, and a useful entity such as host, user, process, or IP.
- Clicking a top-value bar adds that value as a query filter and runs the pivot.
- The schema-aware query guide lists every field and data type, inserts fields at the cursor, and supplies working recipes for the active evidence table.
- The 330 px challenge sidebar scrolls independently and can be hidden with **HIDE CHALLENGES** to give charts and event rows the full viewport.

Supported KQL-style operators are `where`, `search`, `project`, `summarize count() by`, `order by`, and `take`. Text expressions support `contains`, `!contains`, `startswith`, and `endswith`. This is a focused KQL-to-DuckDB learning layer rather than the complete Microsoft Kusto language.

## Publish a lab

1. Copy `admin/lab-spec.template.json` somewhere private and fill in real raw-log paths, challenge answers, hints, MITRE tags, and optional verification SQL.
2. Install `admin/requirements.txt`.
3. Run:

   ```powershell
   python admin/publish_lab.py C:\private\my-lab.json
   ```

The compiler scrubs domains/IPs by default, shifts timestamps, creates Snappy-compressed Parquet files under `data/labs/<lab-id>/`, hashes answers, removes raw paths and plaintext answers, and updates `manifest.json`.

Review the generated public data before committing it. Everything deployed to GitHub Pages is publicly downloadable, even when it is not linked in the interface.

## Build and deploy

```powershell
pnpm install
pnpm build
```

Commit the static project and enable GitHub Pages for the directory containing `index.html`. DuckDB-Wasm, workers, and WebAssembly binaries are vendored under `assets/vendor`; no runtime CDN or backend is required.

For remote Parquet instead of repository-hosted files, put an HTTPS URL in `parquet_sources`. The storage origin must allow CORS and expose `Content-Range`, `Content-Length`, and `Accept-Ranges` so DuckDB can make range requests.

## Manifest challenge fields

- `solution_hash`: SHA-256 of the normalized answer; generated by the admin publisher.
- `required_query_token`: optional query-history token learners must have used.
- `verification_sql`: optional background query that must return a first value greater than zero.
- `hints`: ordered learner hints.

Because this is a fully static client-side lab, hashes and verification queries deter casual answer lookup but cannot provide cryptographic secrecy against a determined learner who can inspect all published assets.
